## What This Project Does (Plain English)

We are building an AI system that watches workout videos and tells you:
1. **Which exercise** is being performed (bench press, biceps curl, shoulder press, or triceps pushdown)
2. **How good your form is** — a score from 0 to 100
3. **How many reps** you completed — and whether each one was correct or sloppy

The user uploads a video (or uses their webcam live), and the app draws a skeleton on top of them and shows all the stats in real time.

## The Two Machines Involved

| Machine | Owner | Role | GPU? |
|---|---|---|---|
| Machine A | Ibrahim | Has the videos, runs the Streamlit app | No GPU (CPU only) |
| Machine B | Friend | Has a GPU | YES — this is where training runs |

**Why this matters:**
- **Pose extraction** (`prepare_data.py`) processes videos and extracts skeleton data. It CAN use GPU if CUDA is available, but it runs on Ibrahim's machine because that's where the videos are. On CPU it takes ~6 minutes per video × 125 videos = ~12 hours.
- **Training** (`train.py`) is the part that MUST run on the GPU machine. Training a deep learning model on CPU would take days. On a decent GPU it takes 30–90 minutes.
- **The handoff**: Ibrahim runs extraction → copies the `data/` folder to the friend's machine → friend trains → copies `checkpoints/best.pt` back to Ibrahim.

## Full Pipeline (What Happens Step by Step)

```
YOUR GYM VIDEOS (MP4 files)
        ↓
[STEP 1 — Ibrahim's machine, CPU OK]
YOLOv8 Pose Extraction
  • Runs every video frame through a pose detection model (yolov8s-pose.pt)
  • Detects 17 body keypoints: nose, eyes, shoulders, elbows, wrists,
    hips, knees, ankles for every frame
  • Saves the result as .npy (numpy array) files
  • Also computes 12 joint angles and velocity/bone length features
  • Output: data/processed/skeletons/*.npy
        ↓
[STEP 2 — Ibrahim's machine]
Manual Labeling (YOU HAVE TO DO THIS)
  • Watch each video, note which frames each rep starts and ends
  • Fill data/annotations/reps.csv with rep_start, rep_end, form_label
  • If you skip this, the system uses automatic rule-based labels (OK but not great)
        ↓
[STEP 3 — Friend's GPU machine, REQUIRED]
Training the Neural Network
  • The model reads the .npy skeleton files in sliding 30-frame windows
  • Learns from thousands of windows what each exercise looks like
  • Simultaneously learns: exercise type, form quality, rep detection
  • Saves the trained model to checkpoints/best.pt
        ↓
[STEP 4 — Ibrahim's machine]
Run the App
  • streamlit run app.py
  • Upload a video or use webcam
  • See skeleton overlay + exercise name + form score + rep count
```

## The Neural Network Architecture (Simplified)

The model takes in 30 frames of skeleton data and outputs 3 predictions:

```
INPUT: 30 consecutive video frames
  • Skeleton: (30 frames × 17 joints × 6 features) = shape (30, 17, 6)
    Features per joint: x position, y position, detection confidence,
                        x velocity, y velocity, bone length to parent
  • Angles: (30 frames × 12 angles) = shape (30, 12)
    12 angles: left/right shoulder, elbow, hip, knee, trunk, ankle

          ↓                    ↓
   ST-GCN Backbone        Angle MLP
   (processes the         (processes the
    skeleton graph)        joint angles)
   Output: (30, 512)      Output: (30, 64)
          ↓                    ↓
          └──────────┬──────────┘
                     ↓
               BiLSTM (reads the sequence of 30 frames)
               2 layers × 256 hidden units × bidirectional
               Output: 512-dimensional feature vector
                     ↓
          ┌──────────┼──────────┐
          ↓          ↓          ↓
    Exercise       Form       Rep
      Head         Head       Head
   (4 classes)  (correct/  (rep in this
                incorrect)  window?)
```

**Why 3 heads?** Because we want to learn exercise, form quality, and rep detection all at once from the same skeleton data. This is called "multi-task learning" — it's more efficient and the tasks help each other.

### STEP 2 — Manual Labeling (Ibrahim does this while extraction runs or after)

**First, generate the template:**
```bash
# Run AFTER prepare_data.py completes (needs splits to exist):
python scripts/build_annotation_template.py
# Creates: data/annotations/reps_template.csv
```

**Then open `data/annotations/reps_template.csv` in Excel or any spreadsheet.**

It looks like this (one row per video, pre-filled with the exercise name):
```
video,exercise,rep_start,rep_end,form_label,mistake_type
bench_press_video1,bench_press,,,,none
bench_press_video2,bench_press,,,,none
biceps_video1,biceps,,,,none
...
```

**You need to fill in:**
- `rep_start` — frame number where the rep STARTS (the moment movement begins)
- `rep_end` — frame number where the rep ENDS (back to starting position)
- `form_label` — `correct` or `incorrect`
- `mistake_type` — optional, e.g. `too_fast`, `incomplete_range`, `none`

**How to find frame numbers:**
Open the video in VLC → View → Advanced Controls → use the frame-by-frame button.
Each second = ~30 frames. A rep that takes 2 seconds starts at say frame 30 and ends at frame 90.

**ONE video can have MULTIPLE rows** (one per rep):
```csv
bench_press_video1,bench_press,45,92,correct,none
bench_press_video1,bench_press,110,160,incorrect,incomplete_range
bench_press_video1,bench_press,190,240,correct,none
```

**Save the file as `data/annotations/reps.csv`** (overwrite the empty one).

**Validate your labels before training:**
```bash
python scripts/validate_annotations.py \
  --annotations data/annotations/reps.csv \
  --splits_dir data/splits \
  --skeleton_dir data/processed/skeletons
```

> **If you don't label anything:** The system will still train using automatic rule-based labels generated from the joint angles. This is called "hybrid mode" and it works, but manual labels produce better results.

**What training does:**
- Loads all the .npy skeleton files
- Creates 30-frame sliding windows (with stride 15) from each video
- Feeds each window through the model
- Computes loss (how wrong the predictions are) across all 3 heads
- Adjusts the model weights to be less wrong (this is "learning")
- Does this for up to 200 epochs (one epoch = seeing all training data once)
- Saves the best model to `checkpoints/best.pt`
- Saves training curves to `logs/` (view with TensorBoard)

**What you'll see:**

## configs/default.yaml — Key Settings Explained

```yaml
model:
  num_exercises: 4          # Number of exercise classes (we have 4)
  lstm_hidden: 256          # BiLSTM hidden size — bigger = more memory
  dropout: 0.4              # Randomly turn off 40% of neurons during training
                            # This prevents the model from memorizing (overfitting)

data:
  window_size: 30           # Each training sample = 30 consecutive frames (~1 second at 30fps)
  stride: 15                # Slide the window by 15 frames each time (50% overlap)
                            # More overlap = more training samples from same video
  exercises:                # The 4 exercise names — ORDER MATTERS (they are class indices 0,1,2,3)
    - bench_press           # index 0
    - biceps                # index 1
    - shoulder_press        # index 2
    - triceps               # index 3
  labeling:
    source: hybrid          # "hybrid" = use manual labels if available, rules otherwise
    fallback_to_rules: true # If no manual label for a video, auto-generate using form_rules.py

training:
  epochs: 200               # Maximum training rounds (stops early if not improving)
  batch_size: 32            # How many windows to process at once (higher = faster but needs more GPU RAM)
  learning_rate: 0.001      # How fast the model learns (too high = unstable, too low = slow)
  early_stopping_patience: 25  # Stop if val_loss doesn't improve for 25 epochs in a row
  label_smoothing: 0.1      # Prevent the model from being overconfident

pose:
  model_size: s             # YOLOv8 size: n (tiny/fast) < s (balanced) < m < l < x (accurate/slow)
  confidence_threshold: 0.5 # Ignore detections below 50% confidence
  smoothing_alpha: 0.35     # EMA smoothing — reduces keypoint jitter between frames
```

## How the Labeling System Works (Hybrid Mode)

For each video, the training pipeline asks:
> "Do we have human labels for this video in reps.csv?"

```
YES → Use human labels (rep_start, rep_end, form_label from reps.csv)
NO  → Run segment_reps() on the joint angles to auto-detect reps
       + Run score_form() to auto-score form quality
       → Use those as labels
```

Either way, the labels get converted to per-window format:
- A 30-frame window that **overlaps a rep** → `rep_label = 1`
- A 30-frame window that **doesn't overlap a rep** → `rep_label = 0`
- `form_label` = the form score of the rep with most overlap in that window

## Troubleshooting

| Problem | Fix |
|---|---|
| `prepare_data.py` crashes mid-way | Run again with `--resume` flag — skips already processed videos |
| `torch.cuda.is_available()` returns False | Reinstall PyTorch with CUDA (see STEP 0) |
| Training loss doesn't decrease | Check if batch_size fits in GPU RAM; try `batch_size: 16` |
| `ERROR: No data to create splits from!` | Video directory path is wrong, or folder names don't match config |
| `WARNING: Unknown exercise directory` | Folder name doesn't match config exercises list |
| App won't open webcam | Another app is using the camera; close it |
| MP4 won't play in browser | Re-encode: `python scripts/inference.py --video x.mp4 --output x_fixed.mp4` |
| `Missing rep head in checkpoint` | Old checkpoint without rep head — retrain or resume |
| Out of GPU memory during training | Reduce `batch_size` in configs/default.yaml (try 16, then 8) |
